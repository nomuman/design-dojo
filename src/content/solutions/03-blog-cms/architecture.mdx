# 模範解答：アーキテクチャ・技術選定

## 推奨技術スタック

| 層 | 採用技術 | 候補 | 採用理由 |
|---|---|---|---|
| フロント | Next.js App Router | Remix, Nuxt | ファイルベースルーティング、Server Componentで初期データ取得がシンプル |
| 認証 | カスタムJWT | Auth.js, Clerk | 学習目的でJWTの仕組みを理解するため。実務ではAuth.jsやClerkを推奨 |
| DB | PostgreSQL | MySQL, SQLite | enum型、配列型、部分インデックス等、複雑な制約が必要なため |
| ORM | Prisma | Drizzle, TypeORM | マイグレーション管理とリレーション表現力がチーム開発に有利 |
| バリデーション | Zod | Yup, Joi | TypeScriptとの親和性が最高。APIスキーマとフロントスキーマの共有可能 |
| 画像ストレージ | Cloudflare R2 | AWS S3, Supabase Storage | S3互換、コストが低い、Cloudflare CDNとの連携が容易 |
| CDN | Cloudflare | CloudFront, Fastly | グローバルエッジネットワーク、レート制限・WAFが組み込み |
| セキュリティ | HttpOnly Cookie | localStorage | refreshTokenはXSS攻撃を防ぐためHttpOnly Cookieで管理 |

## 重要な設計判断

### 公開APIと管理APIの分離：なぜ別エンドポイントか
同じ「記事取得」でも、以下の理由でエンドポイントを分離すべきです：

| 観点 | 公開API | 管理API |
|------|---------|---------|
| 認証 | 不要 | 必須 |
| レスポンス内容 | タイトル・本文・著者名等 | + ステータス・公開日・編集履歴等 |
| アクセス制御 | `published` のみ | 自分の記事（全状態） |
| キャッシュ | CDNキャッシュ可能 | キャッシュ不要（ or 短時間） |
| レート制限 | 厳しい（100req/min） | 緩い（ or なし） |

**分離しない場合の問題**:
- 1つのエンドポイントで「認証の有無」「レスポンスの違い」を分岐すると、コードが複雑になる
- CDNキャッシュの設定が困難（認証あり/なしでキャッシュ戦略が異なる）
- 将来的に「公開APIを外部開発者に提供」する際に、管理APIと混在してセキュリティリスクが増大

### 画像アップロード：なぜプリサインドURL方式か
画像アップロードの方式比較：

| 方式 | メリット | デメリット | 結論 |
|------|---------|-----------|------|
| **サーバー経由** | 認証・バリデーションが容易 | サーバー負荷が高い、帯域コストが増大 | ❌ |
| **クライアント直接（認証情報付き）** | サーバー負荷が低い | クライアントにAWS認証情報を露出、セキュリティリスク | ❌ |
| **プリサインドURL** | サーバー負荷が低い、認証情報を露出しない、有効期限で制限可能 | 実装が少し複雑 | ✅ |

**プリサインドURLのフロー**:
1. クライアントが「アップロード用URLを発行して」とAPIにリクエスト（認証必要）
2. APIがS3/R2プリサインドURLを生成して返却（有効期限: 5分）
3. クライアントがそのURLに対して画像をPUT
4. アップロード完了後、クライアントが記事保存APIを呼び出し、画像URLを含める

### なぜ `middleware.ts` でレート制限を行うのか
公開APIは認証不要のため、悪意のあるユーザーからのDDoS攻撃やスクレイピングのリスクがあります。

Next.js の `middleware.ts` でレート制限を行う理由：
- API Routes の前段でリクエストを遮断できる（無駄な処理を実行しない）
- IPアドレスベースで制限可能
- **ただし**: 本番環境では**WAF（Web Application Firewall）**や**CDNレベルでのレート制限**（Cloudflare Rate Limiting等）も併用すべき

### 権限管理：なぜ RBAC を採用するのか
権限管理の方式比較：

| 方式 | メリット | デメリット | 結論 |
|------|---------|-----------|------|
| **フラグ制御**（`is_admin` boolean） | シンプル | 権限が増えるとカラムが増殖 | ❌ |
| **RBAC**（role enum） | シンプル、一般的 | 細かい権限には不十分 | ✅ 現状の要件なら十分 |
| **ABAC**（属性ベース） | 細かい権限設定が可能 | 複雑、実装コストが高い | △ 将来的な拡張 |

現状の要件（admin / editor / reader）では **RBAC** で十分です。ただし、将来的に「特定の記事だけeditorに編集権限を与える」などの細かい制御が必要になった場合は、**権限テーブル**（`Permission` テーブル）の導入を検討してください。
