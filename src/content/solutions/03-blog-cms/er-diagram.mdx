## 設計意図

### 記事の状態管理：なぜ enum + published_at か
記事の状態を管理する方法として、以下の選択肢があります：

| 設計 | メリット | デメリット | 結論 |
|------|---------|-----------|------|
| `is_published` + `is_draft`（2フラグ） | シンプル | 矛盾した状態（両方true）が作れる | ❌ |
| `published_at`（datetime） | 日時情報を持てる | `archived`（非公開）を表現できない | ❌ |
| `status` enum + `published_at` | 状態を明確に管理、日時も保持 | カラム数が増える | ✅ |

**enum の値と意味**:
- `draft`: 下書き（著者のみ閲覧可）
- `pending`: 公開申請中（editorがadminの承認を待っている）
- `published`: 公開済み（読者が閲覧可）
- `archived`: 非公開（過去に公開されていたが、現在は非公開）

**なぜ `published_at` も持つのか？**
- SEO: 記事の初回公開日を検索エンジンに伝える
- ソート: 「公開日順」でソートする際に使用
- 履歴: `archived` 後も「いつ初めて公開されたか」を記録

### カテゴリの階層構造：なぜ自己参照か
カテゴリを階層構造にするための設計比較：

| 設計 | メリット | デメリット | 結論 |
|------|---------|-----------|------|
| **カラム追加**（`parent_id`, `grandparent_id`） | シンプル | 深さが固定（柔軟性なし） | ❌ |
| **自己参照**（`parent_id`） | 任意の深さ、柔軟 | 再帰クエリが必要 | ✅ |
| **閉包テーブル**（Closure Table） | 階層の取得が高速 | テーブルが増える、実装が複雑 | △ 大規模向け |

現状の要件では**自己参照**で十分です。再帰クエリはPrismaやPostgreSQLのCTE（Common Table Expressions）で実現可能です。

### コメントの階層構造：なぜ `parent_id` か
階層コメント（返信）を実現するため。`parent_id` が `null` なら記事への直接コメント、値があれば別コメントへの返信です。

**最大階層の制限**:
- アプリケーション層で**最大3階層まで**に制限
- 4階層以上になる場合は、フラットな表示（例: `@username への返信」）にする

**無限階層を防ぐ理由**:
- UIが複雑になり、インデントが画面幅を超える
- パフォーマンス劣化（再帰クエリが必要）

### なぜ `ARTICLE` と `TAG` を中間テーブルで紐付けるのか
1つの記事に複数のタグを付け、1つのタグが複数の記事に付く**多対多**の関係を表現するため。

**複合主キー**:
```sql
CREATE TABLE ArticleTag (
    article_id UUID REFERENCES Article(id) ON DELETE CASCADE,
    tag_id UUID REFERENCES Tag(id) ON DELETE CASCADE,
    PRIMARY KEY (article_id, tag_id)
);
```

- `ON DELETE CASCADE` で記事またはタグが削除された場合、紐付けも自動削除
- 複合主キーで「同じ記事に同じタグを重複して付けられない」をDBレベルで保証

### なぜ `USER` に `role` enum を持つのか
権限管理をシンプルにするため。Role-Based Access Control（RBAC）の基本形です。

**権限の対応表**:
| アクション | admin | editor | reader |
|-----------|-------|--------|--------|
| 記事作成 | ✅ | ✅ | ❌ |
| 記事編集（自分） | ✅ | ✅ | ❌ |
| 記事編集（他人） | ✅ | ❌ | ❌ |
| 記事公開 | ✅ | 承認必要 | ❌ |
| コメント投稿 | ✅ | ✅ | ✅ |
| コメント承認 | ✅ | ✅ | ❌ |
| カテゴリ管理 | ✅ | ❌ | ❌ |

**注意**: 複雑な権限管理が必要になった場合（例: 「特定の記事だけeditorに編集権限を与える」）、`role` enum だけでは不十分になります。その場合は**別途権限テーブル**を検討してください。

## 模範解答のポイント

### なぜこれが模範解答か
本設計が「良い設計」と言える理由：

1. **コンテンツのライフサイクルを明確にしている**: `draft` → `pending` → `published` → `archived` の流れが、現実の編集・承認・公開ワークフローを正確に表現している
2. **SEOと履歴の両立**: `published_at` を残すことで、SEOに必要な「初回公開日」を保持しつつ、`archived` で非公開にできる
3. **権限の階層が現実的**: `admin` / `editor` / `reader` の3層は、小〜中規模のCMSに最も一般的で実用的な権限設計
4. **コメントのモデレーション**: 承認制 + 階層構造により、コミュニティの質を保ちながら、対話を促進できる

### 初学者が陥りやすい間違いとの対比

| 項目 | ❌ 初学者が陥りがち | ✅ 模範解答 |
|------|------------------|-----------|
| **記事状態管理** | `is_published` boolean だけ | `status` enum + `published_at` datetime |
| **カテゴリ階層** | `parent_id` + `grandparent_id`（深さ固定） | 自己参照（`parent_id`）で任意の深さに対応 |
| **コメント構造** | `article_id` + `reply_to_comment_id`（フラット） | `parent_id` で階層構造 + 最大3階層制限 |
| **権限管理** | `is_admin` boolean | `role` enum（RBAC） |
| **タグ付け** | `ARTICLE.tags`（配列カラム） | `ARTICLE_TAG` 中間テーブル |
| **画像管理** | サーバーに直接アップロード | プリサインドURLで直接S3/R2へ |

### さらなる深掘り（上級者向け）

#### 記事の改訂版管理（Revision）
現状の設計では、公開済み記事を編集すると「上書き」されますが、より厳密な履歴管理が必要な場合：

```sql
CREATE TABLE ArticleRevision (
  id UUID PRIMARY KEY,
  article_id UUID NOT NULL REFERENCES Article(id),
  title VARCHAR(200) NOT NULL,
  content TEXT NOT NULL,
  summary TEXT,
  cover_image_url VARCHAR(500),
  created_by UUID NOT NULL REFERENCES User(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  change_note VARCHAR(500)  -- "誤字修正" 等の変更理由
);
```

**メリット**:
- 過去のバージョンにいつでも戻せる
- 「誰がいつ何を変更したか」完全に追跡可能
- A/Bテスト（過去のバージョンと新バージョンの比較）が可能

**デメリット**:
- テーブルが増え、ストレージ容量が増大
- 記事取得時に「最新バージョン」を取得するクエリが複雑に

**結論**: ニュースサイトや法律文書等、履歴が重要な場合に採用。一般的なブログでは `updated_at` で十分なことが多い。

#### カテゴリの階層構造：自己参照 vs 閉包テーブル
大規模なカテゴリツリー（数百〜数千のカテゴリ）の場合、**閉包テーブル（Closure Table）**が有効：

```sql
CREATE TABLE CategoryPath (
  ancestor_id UUID NOT NULL REFERENCES Category(id),
  descendant_id UUID NOT NULL REFERENCES Category(id),
  depth INT NOT NULL,
  PRIMARY KEY (ancestor_id, descendant_id)
);
```

**データ例**:
| ancestor_id | descendant_id | depth |
|-------------|---------------|-------|
| 技術 | 技術 | 0 |
| 技術 | フロントエンド | 1 |
| 技術 | React | 2 |
| フロントエンド | フロントエンド | 0 |
| フロントエンド | React | 1 |

**メリット**: 「技術カテゴリの全記事（子孫カテゴリ含む）」を取得する際、単純なJOINだけで完了（再帰クエリ不要）。

**デメリット**: カテゴリの追加・削除・移動時に、複数のレコードを更新する必要がある。

**結論**: カテゴリ数が少ない（数十以下）場合は自己参照で十分。数百以上の場合は閉包テーブルを検討。

#### コメントの承認制：ワークフローエンジン
コメントの承認プロセスが複雑になる場合（例: 「editorの承認 → adminの最終承認」等）：

**Inngest** や **Temporal** 等のワークフローエンジンを使うと、複雑な承認フローをコードで表現できます：

```typescript
// Inngestの例
export const commentApprovalWorkflow = inngest.createFunction(
  { id: "comment-approval" },
  { event: "comment/submitted" },
  async ({ event, step }) => {
    // editorへの通知
    await step.run("notify-editor", async () => {
      await sendEmail(event.data.editorId, "新しいコメントが届きました");
    });

    // editorの承認を待つ（最大48時間）
    const editorApproval = await step.waitForEvent("comment/editor-approved", {
      timeout: "48h",
    });

    if (!editorApproval) {
      // タイムアウト → 自動却下
      await step.run("auto-reject", async () => {
        await rejectComment(event.data.commentId, "承認期限切れ");
      });
      return;
    }

    // adminへの通知
    await step.run("notify-admin", async () => {
      await sendEmail(event.data.adminId, "コメントの承認をお願いします");
    });

    // adminの承認を待つ
    const adminApproval = await step.waitForEvent("comment/admin-approved", {
      timeout: "24h",
    });

    // ...
  }
);
```

**結論**: 承認フローが単純（editorまたはadminのどちらかが承認）なら、ステータス管理だけで十分。多段階承認や条件分岐が必要な場合にワークフローエンジンを検討。
