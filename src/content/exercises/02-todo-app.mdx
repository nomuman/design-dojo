# Exercise 02: TODOタスク管理アプリ

## お題
要件定義のみが与えられています。以下の観点から、自分で設計ドキュメントを作成してください。

### 要件（与えられたもの）
ユーザーが個人のタスクを作成・管理でき、他のユーザーと共有できるWebアプリケーション。

1. **ユーザー認証**: メール+パスワード、JWTセッション管理
2. **タスク管理**: 作成・一覧・詳細・編集・削除
   - ソート・フィルタリング対応（ステータス、優先度、期限、カテゴリ）
   - キーワード検索
3. **タスクの状態遷移**: `todo` → `in_progress` → `done`（前の状態にも戻せる）
4. **カテゴリ・ラベル**: 自分専用カテゴリを作成し、タスクに複数紐付け
5. **期限通知**: 期限24時間前に通知。期限切れタスクは自動的に `overdue` フラグ
6. **タスク共有**: 他のユーザーに「閲覧のみ」または「編集可能」で共有
7. **非機能要件**: 一覧レスポンスタイム100ms以内、通知は期限切れから5分以内、未認証は401

## 設計する項目

### 1. フォルダ・ファイル構成
- どのフォルダに何を置くか、なぜそうするか
- API Routes or Server Actions or 別バックエンド？ なぜか
- 通知処理（cron + キュー）のフォルダ構成

### 2. 技術選定（3つ以上比較）
- 通知キュー: Bull vs BullMQ vs AWS SQS vs RabbitMQ
- スケジューラー: node-cron vs node-schedule vs Inngest
- 認証ライブラリ、ORM、バリデーションなど
- 各候補のトレードオフと採用理由

### 3. API設計書
- エンドポイント、HTTPメソッド、リクエスト/レスポンス、エラーパターン
- 検索・フィルタリングのクエリパラメータ設計
- 共有機能のAPI設計

### 4. DB設計（ER図またはテーブル定義）
- エンティティ、リレーション、制約（ユニーク、外部キー）
- タスク共有の中間テーブル設計
- 通知履歴のテーブル設計
- なぜその設計にしたかの理由

## 注意点
- 「AIに聞いてコピペ」ではなく、**まず自分で紙かテキストエディタに書いてみて**から模範解答を見てください
- 細かいコード（コントローラー内のif文レベル）は不要です。設計レベルの判断が重要
- **特に「期限通知」の実装方法**について、複数の選択肢を比較検討してください
