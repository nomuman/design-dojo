# Exercise 03: ブログ/CMS

## お題
要件定義のみが与えられています。以下の観点から、自分で設計ドキュメントを作成してください。

### 要件（与えられたもの）
ユーザーが記事を作成・管理し、他のユーザーが閲覧・コメントできるブログ/CMSアプリケーション。

1. **ユーザー認証**: メール+パスワード、JWTセッション管理
2. **ユーザーロール**: `admin` / `editor` / `reader`
   - `admin`: 全記事の作成・編集・削除・公開が可能
   - `editor`: 自分の記事の作成・編集・削除が可能、公開は `admin` の承認が必要
   - `reader`: 記事の閲覧・コメントのみ可能
3. **記事管理（ライフサイクル）**: 下書き（draft）→ 公開申請（pending）→ 公開（published）→ 非公開（archived）
4. **カテゴリとタグ**: カテゴリは階層構造、タグは多対多
5. **画像アップロード**: 記事本文・カバー画像。外部ストレージ（S3等）に保存
6. **コメント機能**: 承認制、階層コメント（最大3階層）
7. **公開API vs 管理API**: 公開APIは認証不要、管理APIは認証必要
8. **非機能要件**: 画像アップロード3秒以内、記事一覧100ms以内、公開APIはレート制限

## 設計する項目

### 1. フォルダ・ファイル構成
- どのフォルダに何を置くか、なぜそうするか
- 公開APIと管理APIのURL設計
- 画像アップロードのフロー

### 2. 技術選定（3つ以上比較）
- 画像ストレージ: S3 vs Cloudflare R2 vs Supabase Storage
- CDN: Cloudflare vs CloudFront vs Fastly
- 認証ライブラリ、ORM、バリデーションなど
- 各候補のトレードオフと採用理由

### 3. API設計書
- 公開APIと管理APIのエンドポイント設計
- 画像アップロード（プリサインドURL）のAPI設計
- コメント承認・削除のAPI設計
- エラーレスポンス形式

### 4. DB設計（ER図またはテーブル定義）
- 記事の状態管理（draft/pending/published/archived）
- カテゴリの階層構造（自己参照）
- コメントの階層構造（自己参照）
- 権限設計（RBAC）
- なぜその設計にしたかの理由

## 注意点
- 「AIに聞いてコピペ」ではなく、**まず自分で紙かテキストエディタに書いてみて**から模範解答を見てください
- 細かいコード（コントローラー内のif文レベル）は不要です。設計レベルの判断が重要
- **特に「公開API vs 管理APIの分離」と「画像アップロードのフロー」**について、複数の選択肢を比較検討してください
