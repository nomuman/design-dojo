# 要件定義

## 概要
ユーザーが個人のタスクを作成・管理でき、他のユーザーと共有できるWebアプリケーション。

## 機能要件

### 1. ユーザー認証
- メールアドレス + パスワードによるログイン
- JWT（アクセストークン + リフレッシュトークン）によるセッション管理
- パスワードはハッシュ化（bcrypt）して保存

### 2. タスク管理（自分のタスク）
- タスクの作成（タイトル・説明・期限・優先度・カテゴリ）
- タスクの一覧表示（ソート・フィルタリング対応）
- タスクの詳細表示
- タスクの編集（タイトル・説明・期限・優先度・カテゴリ・ステータス）
- タスクの削除

### 3. タスクの状態遷移
- `todo` → `in_progress` → `done`
- 状態は前の状態に戻せる（例：`done` → `in_progress`）
- 各状態変更にタイムスタンプを記録（`started_at`, `completed_at`）

### 4. カテゴリ・ラベル管理
- カテゴリの作成・編集・削除（自分専用）
- タスクに複数のカテゴリを紐付け可能

### 5. 期限通知
- 期限が近づいたタスク（期限24時間前）をメールまたはアプリ内通知
- 期限切れタスクは自動的に `overdue` フラグを立てる（または状態変更はしない）

### 6. タスク共有
- 自分のタスクを他のユーザーに「閲覧のみ」または「編集可能」で共有
- 共有されたタスクは相手のタスク一覧に表示される
- 共有の解除が可能

### 7. 検索・フィルタリング
- キーワード検索（タイトル・説明）
- ステータスでのフィルタ（todo / in_progress / done / all）
- 優先度でのフィルタ（low / medium / high）
- 期限でのソート（期限近い順・遠い順）
- カテゴリでのフィルタ

## 非機能要件
- タスク一覧のレスポンスタイム: 平均100ms以内（ページネーション必須）
- 通知は期限切れになってから**最大5分以内**に配信されること
- 認証なしのリクエストは401を返す
- refreshToken は XSS 対策のため HttpOnly Cookie で管理する

## 設計のポイント（ヒント）

### 期限通知の実装
期限切れ通知を**フロントエンドのポーリング**で実装すると、ユーザー数が増えた際にサーバー負荷が急増します。正しいアプローチは：

1. **バックエンドの cron ジョブ**（例：node-cron、Bull Queue + Redis）で期限切れタスクを定期的にスキャン
2. 期限が近づいたタスクを**通知キュー**（Redis List、RabbitMQ、AWS SQS等）に投入
3. 別プロセス（ワーカー）がキューを消費して通知（メール送信、プッシュ通知等）

**なぜcron + キューか？**
- ポーリングは無駄なリクエストが発生し、スケールしない
- DBトリガーはビジネスロジックをDBに閉じ込め、テスト・保守が困難になる
- cron + キューは「処理の分散」と「再試行（リトライ）」が容易

### タスク共有のDB設計
「タスクを共有する」機能を実装する際、以下のような設計は避けるべきです：

- **避ける**: `TASK` テーブルに `shared_with_user_id`（単一FK）を追加 → 複数人に共有できない
- **避ける**: `USER.shared_task_ids`（配列） → 正規化違反、検索が遅い、整合性をDBで保てない

**正しい設計**: `TASK_SHARE` 中間テーブルを作成
```
TASK_SHARE
- task_id (FK)
- user_id (FK)  -- 共有先ユーザー
- permission (enum: view / edit)
- created_at
```

**なぜ中間テーブルか？**
- 1つのタスクを複数ユーザーに共有できる
- 権限（view / edit）を柔軟に設定できる
- 共有解除もレコード削除だけでシンプル
- DBレベルで整合性（外部キー制約）を保てる

## 用語集（Glossary）

### Cron（クーロン）
Unix系OSで標準的に使われるジョブスケジューラ。「指定した時間に、指定したコマンドを実行する」仕組みです。Node.jsでは `node-cron` ライブラリで同様の機能を実現できます。

**例**: `0 9 * * *` → 毎日午前9時に実行

**なぜ使うのか**: 期限切れタスクのチェックや、定時バッチ処理等、時間ベースで自動実行したい処理に必須です。

### キュー（Queue）
「待ち行列」のこと。処理したいタスクを順番に入れて、ワーカー（処理プロセス）が順番に取り出して実行する仕組みです。

**本ケースでの使い方**: 期限が近づいたタスクをキューに入れ、ワーカーがメール送信等の通知処理を非同期で実行します。

**なぜキューを使うのか**: 
- 通知処理が重い場合、APIレスポンスを待たされるのを防げる
- 通知失敗時に再試行（リトライ）が容易
- 大量の通知を順番に処理でき、システムの負荷を分散できる

### Redis
インメモリデータストア。キーバリュー型のデータベースで、非常に高速（ミリ秒以下）に読み書きできます。

**本ケースでの使い方**: Bullキューのバックエンドとして使用。ジョブの待ち行列をRedis上に保持します。

**なぜRedisか**: 
- ディスクではなくメモリに保持するため、キューの操作が極めて高速
- リスト（List）データ型を使えば、キューのプッシュ/ポップが原子操作（競合しない）として実行できる
- 永続化オプションもあり、サーバー再起動後もキューが復元できる

### ワークフロー（Workflow）
タスクが「下書き→レビュー中→承認済み→公開」等のように、状態を順番に遷移していく仕組みです。

**本ケースでの例**: タスクの状態遷移（`todo` → `in_progress` → `done`）が最もシンプルなワークフローです。

### ステートマシン（State Machine）
「状態」と「状態間の遷移」を明確に定義したモデル。ワークフローを形式的に表現する方法です。

**本ケースでの例**:
```
todo --[start]--> in_progress
in_progress --[complete]--> done
done --[reopen]--> in_progress
```

**なぜ考えるのか**: 許可されていない遷移（例: `done` → `todo` の直接遷移を禁止したい場合）を、アプリケーション層で明確に制御できるようになります。

### プリサインドURL（Presigned URL）
クラウドストレージ（S3, R2等）が発行する「一定時間のみ有効な一時的なアップロードURL」。認証情報を含んでおり、そのURLに対してHTTP PUTを送るだけでファイルをアップロードできます。

**有効期限**: 通常5分〜1時間。期限切れ後はアクセスできなくなります。

**なぜ使うのか**: クライアントにAWSの認証情報（アクセスキー・シークレットキー）を直接渡す必要がなく、セキュリティリスクを減らせます。

## 設計の深掘り

### なぜ「期限通知」をフロントエンドのポーリングで実装してはいけないのか
**ポーリング**とは、フロントエンドが定期的にサーバーに「何か変化がありませんか？」と問い合わせる方式です。

**問題点**:
- 1000人のユーザーが1分ごとにポーリングすると、毎分1000リクエストが発生
- 大半のリクエストは「変化なし」の無駄な通信
- サーバー負荷がユーザー数に比例して増大し、スケールしない

**正しいアプローチ（Push型）**:
- サーバー側でcronジョブが期限を監視
- 期限が近づいたら、キューに通知ジョブを投入
- ワーカーが実際の通知処理（メール送信等）を非同期で実行

**メリット**:
- サーバー側の処理なので、ユーザー数に依存しない
- 通知の遅延を制御しやすい（「期限5分前に必ず通知」等）
- 通知失敗時の再試行が容易

### なぜタスク共有で「配列カラム」を使ってはいけないのか
`USER.shared_task_ids` のように配列を使う場合の問題：

1. **検索が遅い**: 「このタスクを誰と共有しているか」を調べる際、全ユーザーの `shared_task_ids` をスキャンする必要がある
2. **整合性が保てない**: DBの外部キー制約が効かないため、「存在しないタスクIDが配列に入る」を防げない
3. **同時更新の競合**: 2人が同時に共有設定を変更すると、配列の更新が競合して一方の変更が失われるリスク

**正しい設計**: `TASK_SHARE` 中間テーブルを使い、 `task_id` + `user_id` で複合主キーを定義する。

### なぜ `due_date` は `DATE` 型ではなく `TIMESTAMP` 型か
`DATE` 型は「日付のみ」（例: `2024-01-20`）を表しますが、`TIMESTAMP` 型は「日付+時刻」（例: `2024-01-20T10:00:00+09:00`）を表します。

**本ケースで `TIMESTAMP` が必要な理由**:
- 「1月20日 午前10時までに提出」という要件に対応するため
- ただし、日付だけで良い場合（「1月20日が期限」）もあるため、要件によって使い分ける

**注意**: PostgreSQLでは `TIMESTAMP WITH TIME ZONE`（`timestamptz`）を使い、タイムゾーンを明示すべきです。

## 具体例シナリオ

### シナリオ1: 期限切れ通知の処理フロー
**状況**: 1月20日 10:00 が期限のタスクがある。現在は1月19日 09:00。

**フロー**:
1. **1月19日 09:00**（期限24時間前）:
   - cronジョブが実行され、`due_date` が「現在時刻+24時間」以内のタスクを検索
   - 該当タスクが見つかる → 通知キューに「期限24時間前通知」ジョブを投入

2. **1月19日 09:05**（ワーカー処理）:
   - ワーカーがキューからジョブを取得
   - メール送信APIを呼び出し、「タスク『○○』の期限が24時間後に迫っています」と送信
   - 送信成功 → `NOTIFICATION_LOG.status = 'sent'` に更新
   - 送信失敗 → `NOTIFICATION_LOG.status = 'failed'` に更新。再試行キューに戻す

3. **1月20日 10:01**（期限切れ後）:
   - cronジョブが実行され、`due_date` を過ぎたタスクを検索
   - 該当タスクの `is_overdue = true` に更新
   - 通知キューに「期限切れ通知」ジョブを投入

4. **1月20日 10:06**（ワーカー処理）:
   - ワーカーがキューからジョブを取得
   - メール送信:「タスク『○○』の期限が過ぎました」と送信

**なぜcron + キューを分けるのか**: cronジョブは「いつ通知すべきか」の判断だけを行い、実際の通知処理はキューに委ねます。これにより、cronジョブの実行時間が短くなり、次のスケジュールへの影響を最小化できます。

### シナリオ2: タスク共有の権限制御
**状況**: ユーザーAがタスク「週次レポート作成」をユーザーBに「閲覧のみ」で共有している。

**フロー**:
1. ユーザーBがタスク一覧を取得 → `GET /api/tasks`
2. サーバーが `TASK` テーブル（ユーザーBが作成したタスク）と `TASK_SHARE` テーブル（ユーザーBが共有されているタスク）を結合して検索
3. ユーザーBに「週次レポート作成」タスクが表示される（`permission: view`）
4. ユーザーBがタスクを編集しようとする → `PATCH /api/tasks/xxx`
5. サーバーが `TASK_SHARE` テーブルを確認 → `permission: view` → **403 Forbidden**
6. エラーレスポンス: `{ error: { code: 'FORBIDDEN', message: 'このタスクは閲覧のみです' } }`

**なぜサーバー側で権限チェックが必要か**: フロントエンドで「編集ボタンを非表示」にしても、悪意のあるユーザーが直接APIを呼び出す可能性があるため、サーバー側での権限チェックは絶対条件です。

### シナリオ3: タスクの状態遷移と履歴
**状況**: タスク「週次レポート作成」の状態遷移

**履歴**:
- `created_at`: 2024-01-15 09:00（作成時）→ status: `todo`
- `started_at`: 2024-01-16 14:30（作業開始）→ status: `in_progress`
- `completed_at`: 2024-01-17 11:00（作業完了）→ status: `done`
- ユーザーが「まだ終わってなかった」と気づく
- `completed_at`: null（リセット）→ status: `in_progress`
- `completed_at`: 2024-01-18 16:00（再完了）→ status: `done`

**なぜ `completed_at` をリセットするのか**: タスクを「再度進行中」に戻す際、過去の完了日時は不要になるため。ただし、`updated_at` は変更のたびに更新されるため、履歴の追跡には役立ちます。

**より厳密な履歴管理が必要な場合**: `TASK_STATUS_HISTORY` テーブルを別途作成し、状態変更の全履歴を記録することを検討してください。
