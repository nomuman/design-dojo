# アーキテクチャ・技術選定

## 技術スタック
- **フロントエンド**: Next.js (App Router)
- **バックエンド**: Next.js API Routes
- **認証**: カスタムJWT（学習目的）
- **DB**: PostgreSQL
- **ORM**: Prisma
- **バリデーション**: Zod
- **セキュリティ**: refreshToken は HttpOnly Cookie で管理
- **通知キュー**: Redis + Bull（または BullMQ）
- **スケジューラー**: node-cron（または Inngest）

## 重要な設計判断

### なぜ Redis + Bull を使うのか
期限通知を実装する際の選択肢と比較：

| 方式 | メリット | デメリット | 結論 |
|------|---------|-----------|------|
| **ポーリング**（フロントエンド） | 実装が簡単 | 無駄なリクエスト、スケールしない | ❌ 避ける |
| **DBトリガー** | DBレベルで自動化 | ビジネスロジックをDBに閉じ込め、保守が困難 | ❌ 避ける |
| **Cron + 直接送信** | シンプル | 大量通知時に主処理をブロック、再試行が困難 | △ 小規模のみ |
| **Cron + Queue** | 処理の分散、再試行、並列処理が容易 | Redis等のインフラが必要 | ✅ 推奨 |

**Bull（または BullMQ）を選ぶ理由**：
- Redisベースの軽量なキューシステム
- ジョブの遅延実行（`delay` オプション）、再試行（`attempts`）、優先度付けが組み込み
- Node.jsエコシステムとの親和性が高く、Prismaと同じプロセス内で動作可能
- ただし、本番環境で本格的なスケジューリングが必要な場合は **Inngest** や **Temporal** の検討も有効

### なぜ `workers/` ディレクトリを分けるのか
Next.js API Routes は HTTP リクエストに応答するため、バックグラウンドでの継続的処理（cron、キュー消費）には不向きです。

`workers/` を分離することで：
- API サーバーとワーカープロセスを**独立してスケール**できる
- ワーカーの障害が API サーバーに影響しない
- ただし、開発環境では ` concurrently ` 等で同時起動する必要がある

### なぜ `NOTIFICATION_LOG` を独立テーブルにするのか
通知履歴を `TASK` にカラムとして持つ場合との比較：

- **TASKにカラム追加**: `notified_at`（datetime）
  - ❌ 通知方法（メール/アプリ内）の区別ができない
  - ❌ 通知失敗時の再試行履歴が残らない
  - ❌ 共有相手への通知に対応できない

- **独立テーブル**: `NOTIFICATION_LOG`
  - ✅ 「誰に」「どの方法で」「成功したか」を完全に追跡
  - ✅ 再試行時に過去の失敗履歴を参照可能
  - ✅ 拡張性：プッシュ通知、Slack通知等への追加が容易
