# 要件定義

## 概要
ユーザーが記事を作成・管理し、他のユーザーが閲覧・コメントできるブログ/CMSアプリケーション。

## 機能要件

### 1. ユーザー認証
- メールアドレス + パスワードによるログイン
- JWT（アクセストークン + リフレッシュトークン）によるセッション管理
- パスワードはハッシュ化（bcrypt）して保存
- **ユーザーロール**: `admin`（管理者）/ `editor`（編集者）/ `reader`（読者）
  - `admin`: 全記事の作成・編集・削除・公開が可能
  - `editor`: 自分の記事の作成・編集・削除が可能、公開は `admin` の承認が必要
  - `reader`: 記事の閲覧・コメントのみ可能

### 2. 記事管理（ライフサイクル）
- 記事の作成（タイトル・本文・サマリー・カバー画像）
- 記事の下書き保存（`draft`）
- 記事の公開申請（`pending`）
- 記事の公開（`published`）
- 記事の非公開（`archived`）
- 記事の編集（公開済み記事の編集は「改訂版」として管理）
- 記事の削除（論理削除）

### 3. カテゴリとタグ
- **カテゴリ**: 階層構造（親カテゴリ → 子カテゴリ）
  - 例: `技術` → `フロントエンド` → `React`
- **タグ**: 記事に複数紐付け可能。階層なし。
- カテゴリとタグは管理者のみ作成・編集・削除可能

### 4. 画像アップロード
- 記事本文内に画像を挿入可能
- カバー画像を設定可能
- 画像は外部ストレージ（S3 / Cloudflare R2 / Supabase Storage）に保存
- 画像URLはCDN経由で配信
- アップロード可能な形式: JPG, PNG, GIF, WebP
- ファイルサイズ上限: 5MB

### 5. コメント機能
- 公開済み記事に対してコメント可能
- コメントは承認制（`admin` / `editor` が承認してから公開）
- コメントに対する返信（階層構造、最大3階層まで）
- 自分のコメントの削除が可能
- 未承認コメントは `pending`、承認済みは `approved`、削除済みは `rejected`

### 6. 検索・フィルタリング
- キーワード検索（タイトル・本文・サマリー）
- カテゴリでのフィルタ
- タグでのフィルタ
- 著者でのフィルタ
- 公開日でのソート（新しい順・古い順）
- 記事一覧はページネーション対応

### 7. 公開API vs 管理API
- **公開API**（認証不要）: 記事一覧・詳細、カテゴリ一覧、タグ一覧、コメント一覧（承認済みのみ）
- **管理API**（認証必要）: 記事作成・編集・削除、コメント承認・削除、カテゴリ・タグ管理

## 非機能要件
- 画像アップロードのレスポンスタイム: 平均3秒以内（5MBの場合）
- 記事一覧のレスポンスタイム: 平均100ms以内（ページネーション必須）
- 公開APIはレート制限（1分間に100リクエスト）
- 認証なしの管理APIリクエストは401を返す
- refreshToken は XSS 対策のため HttpOnly Cookie で管理する

## 設計のポイント（ヒント）

### 記事の状態管理
記事の状態（draft / pending / published / archived）を管理する際、以下のような設計は避けるべきです：

- **避ける**: `is_published`（boolean） + `is_draft`（boolean）の2フラグ
  - 理由: `is_published=true` かつ `is_draft=true` という矛盾した状態が作れてしまう

- **避ける**: `published_at`（datetime）がnullなら下書き、値があれば公開
  - 理由: `archived`（非公開）状態を表現できない。公開後に非公開にしても `published_at` は残る

**正しい設計**: `status`（enum: draft / pending / published / archived） + `published_at`（datetime）の組み合わせ
- `status` で「現在の状態」を管理
- `published_at` で「初回公開日時」を記録（SEOやソートに使用）
- `archived` の場合は `published_at` は残る（過去に公開されていた記録）

### 公開API vs 管理APIの分離
同じ「記事取得」でも、公開APIと管理APIは**別のエンドポイント**にすべきです。

- **公開API**: `GET /api/public/articles` - 認証不要、`published` のみ返す
- **管理API**: `GET /api/articles` - 認証必要、自分の記事（全状態）を返す

**なぜ分けるのか？**
- 認証の有無、レスポンスの内容、アクセス制御の粒度が異なる
- 公開APIはCDNキャッシュやレート制限の対象
- 管理APIは認可（authorization）のロジックが複雑になる
- 将来的に公開APIを外部開発者に提供する場合、別管理が容易

### 画像アップロードのフロー
クライアントから直接S3にアップロードする「プリサインドURL」方式が一般的です。

**フロー**:
1. クライアントが「アップロード用URLを発行して」とAPIにリクエスト
2. APIがS3プリサインドURLを生成して返却（有効期限: 5分）
3. クライアントがそのURLに対して画像をPUT
4. アップロード完了後、クライアントが記事保存APIを呼び出し、画像URLを含める

**なぜプリサインドURLか？**
- サーバーを経由せず直接S3にアップロードできるため、サーバー負荷が軽減される
- クライアントにAWS認証情報を渡す必要がない
- アップロード可能なファイルサイズ・形式をURL発行時に制限できる

## 用語集（Glossary）

### RBAC（Role-Based Access Control）
「ロール（役割）」に基づいてアクセス権限を管理する方式。`admin` / `editor` / `reader` 等のロールを定義し、各ロールに対して「何ができるか」を設定します。

**なぜ使うのか**: ユーザー一人ひとりに権限を設定するのではなく、ロールにまとめて設定することで、管理が簡単になります。例えば、「editorを全員readonlyに変更」もロールの定義を変えるだけで一括適用できます。

**注意点**: ロールだけでは「特定の記事だけeditorに編集権限を与える」といった細かい制御ができません。その場合は**ABAC（Attribute-Based Access Control）**や**権限テーブル**を検討してください。

### CDN（Content Delivery Network）
画像やCSS、JavaScript等の静的ファイルを、世界中のエッジサーバー（ユーザーの近くにあるサーバー）から配信する仕組みです。

**なぜ使うのか**: 
- サーバーから直接配信するよりも高速（レイテンシ削減）
- サーバーの帯域コストを削減
- DDoS攻撃等からサーバーを保護（WAF機能）

**本ケースでの使い方**: アップロードされた画像をCloudflareやCloudFront経由で配信し、ブログ記事の読み込み速度を高速化します。

### プリサインドURL（Presigned URL）
クラウドストレージ（S3, R2等）が発行する「一定時間のみ有効な一時的なアップロードURL」。認証情報を含んでおり、そのURLに対してHTTP PUTを送るだけでファイルをアップロードできます。

**有効期限**: 通常5分〜1時間。期限切れ後はアクセスできなくなります。

**なぜ使うのか**: クライアントにAWSの認証情報（アクセスキー・シークレットキー）を直接渡す必要がなく、セキュリティリスクを減らせます。

### レート制限（Rate Limiting）
単位時間あたりのリクエスト数を制限する仕組み。「1分間に100リクエストまで」等の制約を設けます。

**なぜ使うのか**: 
- DDoS攻撃やスクレイピングからサーバーを保護
- リソースの公平な配分（特定のユーザーが独占しないように）
- コスト管理（API利用料金がリクエスト数に応じて増える場合）

**本ケースでの使い方**: 公開API（認証不要）に対してIPアドレスベースでレート制限を設け、悪意のある大量アクセスを防ぎます。

### 論理削除 vs 物理削除
| | 論理削除 | 物理削除 |
|---|---|---|
| **仕組み** | `status = 'deleted'` 等のフラグを立てる | `DELETE` 文でレコードを削除 |
| **データ復元** | 可能（フラグを元に戻す） | 不可能（バックアップが必要） |
| **ディスク容量** | 増える | 減る |
| **クエリ速度** | インデックスを貼る必要あり | 問題なし |
| **用途** | 監査・履歴が必要な場合 | 一時的なデータ（ログ等） |

**本ケースでの結論**: 記事データは「誰がいつ編集/削除したか」の監査が必要なため、論理削除を採用します。

### マークダウン（Markdown）
記事本文を書くための軽量なマークアップ言語。`# 見出し`、`**太字**`、`- リスト` 等の簡易的な記法で、HTMLに変換できます。

**なぜ使うのか**: リッチテキストエディタ（Word等）よりも軽量で、プログラマにとって馴染み深い記法です。ただし、一般ユーザー向けにはWYSIWYGエディタ（Markdownを裏で使う）を提供するのが親切です。

### SEO（Search Engine Optimization）
検索エンジン（Google等）で上位に表示されるように、Webページを最適化する技術。

**本ケースでの考慮点**:
- 記事のURLにスラッグ（`my-first-post` 等の読みやすい識別子）を使用
- `<title>` タグと `<meta name="description">` を動的に設定
- 構造化データ（JSON-LD）で記事のタイトル・著者・公開日を検索エンジンに伝える

## 設計の深掘り

### なぜ「公開API vs 管理API」を分離するのか
同じ「記事取得」でも、以下の理由でエンドポイントを分離すべきです：

| 観点 | 公開API | 管理API |
|------|---------|---------|
| 認証 | 不要 | 必須 |
| レスポンス内容 | タイトル・本文・著者名等 | + ステータス・公開日・編集履歴等 |
| アクセス制御 | `published` のみ | 自分の記事（全状態） |
| キャッシュ | CDNキャッシュ可能 | キャッシュ不要（ or 短時間） |
| レート制限 | 厳しい（100req/min） | 緩い（ or なし） |

**分離しない場合の問題**:
- 1つのエンドポイントで「認証の有無」「レスポンスの違い」を分岐すると、コードが複雑になる
- CDNキャッシュの設定が困難（認証あり/なしでキャッシュ戦略が異なる）
- 将来的に「公開APIを外部開発者に提供」する際に、管理APIと混在してセキュリティリスクが増大

### なぜカテゴリは階層構造にするのか
フラットなカテゴリ（階層なし）にすると、以下の問題が生じます：

- **「技術」カテゴリの記事を一覧表示したい場合**: 「フロントエンド」「バックエンド」「インフラ」等の子カテゴリも含めて検索する必要がある
- **URL設計**: `/categories/frontend` と `/categories/backend` が別ページになってしまい、親カテゴリ「技術」のページが意味を持たない

**階層構造のメリット**:
- 親カテゴリのページで、子カテゴリの記事も含めて表示可能
- パンくずリスト（`Home > 技術 > フロントエンド > React`）が自然に実現できる
- ただし、深さは3階層程度に制限すべき（UIが複雑になりすぎるため）

### なぜコメントは承認制にするのか
承認制にしない場合のリスク：

- **スパム**: 自動化された bot が大量の広告コメントを投稿
- **悪意のある内容**: 誹謗中傷、個人情報の漏洩等
- **法的リスク**: 運営者がコメント内容に対して責任を負う可能性がある（各国の法律によって異なる）

**承認制のデメリット**:
- リアルタイム性が失われる（承認待ちの間、コメントが表示されない）
- 運営側の作業負荷が増える

**代替案**: 初回投稿者のみ承認制、過去に承認済みのユーザーは自動承認（信頼ベースのモデル）

## 具体例シナリオ

### シナリオ1: 記事のライフサイクル
**状況**: editorのユーザーAが新しい記事を作成し、公開までの流れ。

**フロー**:
1. **下書き作成**:
   - ユーザーAが `POST /api/articles` を呼び出し、`status: draft`
   - 記事は `ARTICLE` テーブルに保存され、`status = 'draft'`
   - ユーザーAのみ閲覧可能

2. **公開申請**:
   - ユーザーAが `PATCH /api/articles/xxx` を呼び出し、`status: pending`
   - `status = 'pending'` に更新
   - adminに通知（メール or ダッシュボード）

3. **adminの承認**:
   - adminが管理画面で記事を確認
   - `PATCH /api/articles/xxx` を呼び出し、`status: published`
   - `published_at` に現在時刻が自動設定
   - 記事が公開APIから読み込めるようになる

4. **非公開（アーカイブ）**:
   - 数月後、adminが `PATCH /api/articles/xxx` を呼び出し、`status: archived`
   - `archived_at` に現在時刻が自動設定
   - 公開APIからは非表示になるが、`published_at` は残る

**なぜ `published_at` を残すのか**: SEOの観点から「いつ初めて公開されたか」は重要な情報です。また、ユーザーのブラウザ履歴やSNSのシェアリンクからアクセスされた場合、適切なメッセージ（「この記事はアーカイブされました」）を表示できます。

### シナリオ2: 画像アップロードのフロー
**状況**: ユーザーが記事本文に画像を挿入したい。

**フロー**:
1. ユーザーがエディタで「画像挿入」ボタンを押す
2. フロントエンドが `POST /api/upload/presigned-url` を呼び出す
   ```json
   {
     "fileName": "diagram.png",
     "fileType": "image/png",
     "fileSize": 2048000
   }
   ```
3. APIが以下を検証:
   - `fileSize <= 5MB` → OK
   - `fileType` が許可リストに含まれる → OK
4. APIがCloudflare R2にプリサインドURLを発行
5. APIがフロントエンドに返却:
   ```json
   {
     "uploadUrl": "https://xxx.r2.cloudflarestorage.com/...",
     "publicUrl": "https://cdn.example.com/uploads/diagram.png",
     "expiresIn": 300
   }
   ```
6. フロントエンドが `uploadUrl` に対して `PUT` リクエストで画像をアップロード
7. アップロード完了後、フロントエンドが記事保存APIを呼び出し、本文に `![図解](https://cdn.example.com/uploads/diagram.png)` を含める

**なぜこのフローが安全か**: サーバーは画像データを一切受け取らないため、サーバーの帯域とストレージを節約できます。また、プリサインドURLには有効期限があるため、URLが漏洩しても長期間悪用されるリスクはありません。

### シナリオ3: コメントの承認フロー
**状況**: readerのユーザーBが公開記事にコメントを投稿。

**フロー**:
1. ユーザーBが `POST /api/public/articles/xxx/comments` を呼び出し:
   ```json
   {
     "content": "素晴らしい記事です！",
     "parentId": null
   }
   ```
2. サーバーがコメントを `COMMENT` テーブルに保存、`status = 'pending'`
3. editorのユーザーA（記事の著者）に通知（メール or ダッシュボード）
4. ユーザーAが管理画面でコメントを確認
5. ユーザーAが `PATCH /api/comments/yyy/approve` を呼び出す
6. `status = 'approved'` に更新
7. 公開API（`GET /api/public/articles/xxx/comments`）から返されるコメントリストに含まれる

**なぜ承認制が必要か**: スパムや悪意のあるコメントを防ぎ、法的リスクを減らすため。ただし、承認待ちの間はユーザーBに「コメントは承認待ちです」と表示すべきです。
