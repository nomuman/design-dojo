# 要件定義

## 概要
ユーザーが記事を作成・管理し、他のユーザーが閲覧・コメントできるブログ/CMSアプリケーション。

## 機能要件

### 1. ユーザー認証
- メールアドレス + パスワードによるログイン
- JWT（アクセストークン + リフレッシュトークン）によるセッション管理
- パスワードはハッシュ化（bcrypt）して保存
- **ユーザーロール**: `admin`（管理者）/ `editor`（編集者）/ `reader`（読者）
  - `admin`: 全記事の作成・編集・削除・公開が可能
  - `editor`: 自分の記事の作成・編集・削除が可能、公開は `admin` の承認が必要
  - `reader`: 記事の閲覧・コメントのみ可能

### 2. 記事管理（ライフサイクル）
- 記事の作成（タイトル・本文・サマリー・カバー画像）
- 記事の下書き保存（`draft`）
- 記事の公開申請（`pending`）
- 記事の公開（`published`）
- 記事の非公開（`archived`）
- 記事の編集（公開済み記事の編集は「改訂版」として管理）
- 記事の削除（論理削除）

### 3. カテゴリとタグ
- **カテゴリ**: 階層構造（親カテゴリ → 子カテゴリ）
  - 例: `技術` → `フロントエンド` → `React`
- **タグ**: 記事に複数紐付け可能。階層なし。
- カテゴリとタグは管理者のみ作成・編集・削除可能

### 4. 画像アップロード
- 記事本文内に画像を挿入可能
- カバー画像を設定可能
- 画像は外部ストレージ（S3 / Cloudflare R2 / Supabase Storage）に保存
- 画像URLはCDN経由で配信
- アップロード可能な形式: JPG, PNG, GIF, WebP
- ファイルサイズ上限: 5MB

### 5. コメント機能
- 公開済み記事に対してコメント可能
- コメントは承認制（`admin` / `editor` が承認してから公開）
- コメントに対する返信（階層構造、最大3階層まで）
- 自分のコメントの削除が可能
- 未承認コメントは `pending`、承認済みは `approved`、削除済みは `rejected`

### 6. 検索・フィルタリング
- キーワード検索（タイトル・本文・サマリー）
- カテゴリでのフィルタ
- タグでのフィルタ
- 著者でのフィルタ
- 公開日でのソート（新しい順・古い順）
- 記事一覧はページネーション対応

### 7. 公開API vs 管理API
- **公開API**（認証不要）: 記事一覧・詳細、カテゴリ一覧、タグ一覧、コメント一覧（承認済みのみ）
- **管理API**（認証必要）: 記事作成・編集・削除、コメント承認・削除、カテゴリ・タグ管理

## 非機能要件
- 画像アップロードのレスポンスタイム: 平均3秒以内（5MBの場合）
- 記事一覧のレスポンスタイム: 平均100ms以内（ページネーション必須）
- 公開APIはレート制限（1分間に100リクエスト）
- 認証なしの管理APIリクエストは401を返す
- refreshToken は XSS 対策のため HttpOnly Cookie で管理する

## 設計のポイント（ヒント）

### 記事の状態管理
記事の状態（draft / pending / published / archived）を管理する際、以下のような設計は避けるべきです：

- **避ける**: `is_published`（boolean） + `is_draft`（boolean）の2フラグ
  - 理由: `is_published=true` かつ `is_draft=true` という矛盾した状態が作れてしまう

- **避ける**: `published_at`（datetime）がnullなら下書き、値があれば公開
  - 理由: `archived`（非公開）状態を表現できない。公開後に非公開にしても `published_at` は残る

**正しい設計**: `status`（enum: draft / pending / published / archived） + `published_at`（datetime）の組み合わせ
- `status` で「現在の状態」を管理
- `published_at` で「初回公開日時」を記録（SEOやソートに使用）
- `archived` の場合は `published_at` は残る（過去に公開されていた記録）

### 公開API vs 管理APIの分離
同じ「記事取得」でも、公開APIと管理APIは**別のエンドポイント**にすべきです。

- **公開API**: `GET /api/public/articles` - 認証不要、`published` のみ返す
- **管理API**: `GET /api/articles` - 認証必要、自分の記事（全状態）を返す

**なぜ分けるのか？**
- 認証の有無、レスポンスの内容、アクセス制御の粒度が異なる
- 公開APIはCDNキャッシュやレート制限の対象
- 管理APIは認可（authorization）のロジックが複雑になる
- 将来的に公開APIを外部開発者に提供する場合、別管理が容易

### 画像アップロードのフロー
クライアントから直接S3にアップロードする「プリサインドURL」方式が一般的です。

**フロー**:
1. クライアントが「アップロード用URLを発行して」とAPIにリクエスト
2. APIがS3プリサインドURLを生成して返却（有効期限: 5分）
3. クライアントがそのURLに対して画像をPUT
4. アップロード完了後、クライアントが記事保存APIを呼び出し、画像URLを含める

**なぜプリサインドURLか？**
- サーバーを経由せず直接S3にアップロードできるため、サーバー負荷が軽減される
- クライアントにAWS認証情報を渡す必要がない
- アップロード可能なファイルサイズ・形式をURL発行時に制限できる
