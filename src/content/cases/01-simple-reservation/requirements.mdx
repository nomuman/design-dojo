# 要件定義

## 概要
ユーザーが施設や設備を日時指定で予約できるWebアプリケーション。

## 機能要件
1. **ユーザー認証**
   - メールアドレス + パスワードによるログイン
   - JWT（アクセストークン + リフレッシュトークン）によるセッション管理
   - パスワードはハッシュ化（bcrypt）して保存

2. **予約管理**
   - 予約の作成（日時・施設・目的を指定）
   - 予約の一覧表示（自分の予約のみ閲覧可能）
   - 予約の詳細表示
   - 予約のキャンセル（開始時刻前のみ可能）
   - 予約の変更（開始時刻前のみ可能）

3. **施設管理**
   - 施設の一覧表示（全ユーザー閲覧可能）
   - 施設の詳細表示（写真、説明、定員）

## 非機能要件
- 同時刻・同施設の重複予約を防ぐ（DBの一意制約 + アプリケーション層での範囲重複チェック）
- レスポンスタイム: API平均200ms以内
- 認証なしのリクエストは401を返す
- refreshToken は XSS 対策のため HttpOnly Cookie で管理する

## 用語集（Glossary）

### JWT（JSON Web Token）
認証情報を安全にやり取りするためのトークン形式。ヘッダー・ペイロード・署名の3部分から構成され、Base64URLエンコードされた文字列です。

**なぜ使うのか**: サーバーがセッション状態を持たなくても（ステートレス）、クライアントが持つトークンだけでユーザーを識別できるため、スケールが容易です。

**注意点**: JWTは「改ざんを防ぐ」仕組みですが、「暗号化する」仕組みではありません。機密情報（パスワード等）をペイロードに含めないでください。

### アクセストークン vs リフレッシュトークン
| | アクセストークン | リフレッシュトークン |
|---|---|---|
| **有効期限** | 短い（15分〜1時間） | 長い（7日〜30日） |
| **用途** | APIリクエスト時に使用 | アクセストークンの再発行 |
| **保存場所** | メモリ（推奨）またはlocalStorage | **HttpOnly Cookie**（必須） |
| **漏洩時の影響** | 限定的（短時間で無効化） | 重大（長期間有効） |

**なぜ2種類必要か**: アクセストークンが漏洩した場合、有効期限が短いため被害を最小化できます。リフレッシュトークンは長期間有効なため、より安全なCookieに保存します。

### bcrypt
パスワードハッシュ化のためのライブラリ。単なるハッシュ関数（SHA-256等）ではなく、「ソルト（ランダムな文字列）」+「ストレッチング（繰り返しハッシュ）」を組み合わせた仕組みです。

**なぜ使うのか**: 同じパスワードでも毎回異なるハッシュ値になり、レインボーテーブル攻撃（事前計算されたハッシュ表を使った攻撃）を防げます。

### HttpOnly Cookie
JavaScriptからアクセスできないCookie。`Set-Cookie` ヘッダーに `HttpOnly` 属性を付与することで実現します。

**なぜ使うのか**: XSS（クロスサイトスクリプティング）攻撃で悪意のあるスクリプトがCookieを盗むのを防ぎます。`document.cookie` では読み取れません。

### 一意制約（Unique Constraint）
DBのカラム（または複数カラムの組み合わせ）に「重複を許可しない」ルールを設ける仕組みです。

**例**: `CREATE UNIQUE INDEX idx ON Reservation(facility_id, start_time)` → 同じ施設で同じ開始時刻の予約を2件作れなくなります。

### 楽観ロック vs 悲観ロック
| | 楽観ロック | 悲観ロック |
|---|---|---|
| **仕組み** | 更新時に「読んだ後にデータが変わっていないか」チェック | 読み込み時に行をロックし、他のトランザクションをブロック |
| **ユースケース** | 衝突が少ない場合（SNSのいいね等） | 衝突が多い場合（予約の重複防止、在庫管理等） |
| **デメリット** | 同時更新が多いと失敗・リトライが増える | ロック中は他の操作が待たされる（デッドロックのリスク） |

**本ケースでの結論**: 予約の重複防止には「悲観ロック（`SELECT FOR UPDATE`）+ DBの一意制約」が適切です。楽観ロックは同時更新の失敗を防げません。

## よくある誤解・トラップ

### ❌ 誤解1: 「JWTをlocalStorageに保存すればシンプル」
**問題**: XSS攻撃を受けると `localStorage.getItem('token')` でトークンが盗まれます。特にリフレッシュトークンが盗まれると、攻撃者は長期間なりすませます。

**正解**: アクセストークンはメモリ（React Context等）に、リフレッシュトークンはHttpOnly Cookieに保存します。

### ❌ 誤解2: 「DBの一意制約だけで重複予約を防げる」
**問題**: `facility_id + start_time` の一意制約は**開始時刻の完全一致**のみ防ぎます。10:00-11:00 と 10:30-11:30 のような「時間帯が重複する」予約は防げません。

**正解**: DB制約 + アプリケーション層での範囲重複チェック（`start_time < 新終了時刻 AND end_time > 新開始時刻`）が必要です。

### ❌ 誤解3: 「パスワードをSHA-256でハッシュ化すれば安全」
**問題**: SHA-256は高速に計算できるため、GPUを使った総当たり攻撃（ブルートフォース）に脆弱です。同じパスワードは常に同じハッシュ値になり、レインボーテーブル攻撃も有効です。

**正解**: bcrypt, Argon2, PBKDF2 等の「ストレッチング（繰り返し計算）」を行う専用のパスワードハッシュ関数を使用します。

### ❌ 誤解4: 「予約キャンセルはDELETEで物理削除すればシンプル」
**問題**: 物理削除（`DELETE`）すると「誰がいつキャンセルしたか」の履歴が完全に失われます。トラブル対応や統計分析が不可能になります。

**正解**: `status` カラムを持たせ、論理削除（`status = 'cancelled'`）にします。必要に応じて `cancelled_at` も記録します。

## 具体例シナリオ

### シナリオ1: 予約作成時の重複チェック
**状況**: ユーザーAとユーザーBが、同時に「会議室Aの10:00-11:00」を予約しようとしています。

**流れ**:
1. ユーザーAが予約画面を開き、10:00-11:00を選択（まだDBには書き込まれていない）
2. ユーザーBも同じ画面を開き、10:00-11:00を選択（まだDBには書き込まれていない）
3. ユーザーAが「予約」ボタンを押す → アプリケーション層で重複チェック → OK → DBにINSERT → 成功
4. ユーザーBが「予約」ボタンを押す → アプリケーション層で重複チェック → ユーザーAの予約が既にある → エラー「指定された時間帯は既に予約されています」

**なぜこれで防げるのか**: アプリケーション層のチェックは「読み込み時」ではなく「書き込み直前」に行うため、他のユーザーの予約を考慮できます。さらにDBの一意制約が二重の安全網になります。

### シナリオ2: JWTトークンの更新
**状況**: ユーザーのアクセストークンの有効期限が切れました。

**流れ**:
1. フロントエンドがAPIを呼び出す → `Authorization: Bearer <accessToken>`
2. APIが「トークン期限切れ（401）」を返す
3. フロントエンドが `/api/auth/refresh` を呼び出す → Cookieに含まれるリフレッシュトークンを送信
4. サーバーがリフレッシュトークンを検証 → 新しいアクセストークンを発行
5. フロントエンドが新しいアクセストークンを使って元のAPIリクエストを再試行

**注意点**: リフレッシュトークンも定期的にローテーション（新しいトークンに交換）すべきです。これにより、リフレッシュトークンが盗まれても影響期間を限定できます。

### シナリオ3: キャンセル済み予約の再利用
**状況**: ユーザーが10:00-11:00の予約をキャンセルし、その後同じ時間帯を再度予約したい。

**流れ**:
1. ユーザーが予約をキャンセル → `status = 'cancelled'` に更新（物理削除しない）
2. ユーザーが同じ時間帯を再度予約 → アプリケーション層で重複チェック
3. 重複チェックの条件: `status = 'confirmed' AND 時間帯が重複` → キャンセル済みは対象外なのでOK
4. 新しい予約が作成される

**なぜ物理削除しないのか**: 「誰がいつキャンセルしたか」の履歴を残すため。ただし、GDPR等の個人情報保護規制に従い、完全な個人情報削除が必要な場合は別途「アカウント削除API」で対応します。
